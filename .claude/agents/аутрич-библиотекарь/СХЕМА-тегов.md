# Схема тегов библиотеки

Единый язык, которым описываем каждую цепочку в реестре и каждого победителя. Библиотекарь матчит карточку проекта по этим осям.

## Оси тега

1. **Ниша** — отрасль/рынок клиента (сертификация, инжиниринг производств, ювелирка, базы B2B-контактов, реф-контейнеры, бизнес-миссии, AI-автоматизация…).
2. **Тип оффера:**
   - `услуга` — продаём работу/сервис под ключ (разная структура CTA).
   - `мероприятие` — приглашаем на событие с датой/местом/программой.
   - `продукт` — продаём конкретный товар/SaaS.
3. **Awareness аудитории** (по Шварцу): `unaware` / `problem` / `solution` / `product`. Определяет опенер. Антипаттерн bpmsoft: опенер уровня `solution` на `problem`-аудиторию.
4. **ЛПР** — кому письмо (собственник / IT-директор / главбух / маркетолог / закупщик…).
5. **Триггер айсбрекера** — вакансии / дата регламента / сезон / недавняя регистрация / истекающий документ / рост объёмов.
6. **Метрика** (из реестра) — reply % и lead %. Без неё цепочка не победитель.

## Статусы цепочки
- `черновик` — написана, метрики нет.
- `в работе` — отправляется, ждём цифры.
- `победитель` — есть подтверждённая метрика выше базовой по нише.
- `антипаттерн` — отработала ощутимо хуже, держим как урок «не повторять».

## Правило победителя
Цепочка получает статус `победитель` только когда в `реестр-результатов.md` есть строка с реальными цифрами (open/reply/лиды) и она выше базовой. Субъективное «вроде зашло» статуса не даёт.
