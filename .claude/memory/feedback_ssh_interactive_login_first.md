Когда нужно поработать на VPS через терминал — первая команда всегда **простая интерактивная `ssh` без пароля в строке**, например:

```
ssh root@85.137.165.185
```

После неё терминал сам запросит пароль, пользователь введёт его — символы не отображаются, в истории терминала пароль не остаётся. Дальше все диагностические команды идут уже **внутри открытой SSH-сессии** отдельными шагами (тоже через HTML-кнопки копирования).

**Why:** пользователь после диагностики копирует **весь вывод терминала** агенту обратно. Если использовать `sshpass -p '<root_pass>' ssh ...` одной строкой — пароль попадает в вывод/скриншот, и его придётся вычищать вручную. Интерактивный ввод пароля прячет его автоматически. Пользователь явно сказал: «не беспокойся, что я случайно тебе дам пароль» — он сам отвечает за то, что не отдаст, моя задача — формат, который физически не палит пароль.

**How to apply:**
- Триггер: любая работа на VPS парсера (или любом другом сервере), где требуется ssh.
- Шаг 1 в HTML — отдельная кнопка с командой `ssh root@<ip>` (без sshpass, без -p).
- Шаг 2+ — диагностические команды (`systemctl status parser`, `tail -80 /var/log/parser.log` и т.п.), каждая своей кнопкой, рассчитанные на выполнение **уже внутри SSH-сессии**.
- Финальный шаг — `exit` или Ctrl+D.
- Не объединять команды через `ssh root@host '...команда...'` — это вынуждает либо sshpass (палит пароль), либо повторный ввод пароля на каждую команду.
- Связано с [[feedback-parser-vps-no-password]] (пароль агенту не показываем — отсюда же логика, что и в выводе терминала ему делать нечего) и [[feedback-commands-as-html-with-copy]] (формат подачи).
