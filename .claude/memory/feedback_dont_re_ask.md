Если пользователь уже сформулировал задачу (даже общими словами), не возвращаться к ней с уточняющими вопросами в духе «а у тебя есть список?» / «а как ты хочешь?». Действовать на разумных дефолтах и показывать результат-черновик. Пользователь поправит, если что-то не так. Возвращаться с вопросом — только если без ответа физически нельзя ничего сделать (и тогда **один** вопрос, не три).

**Why:** в сессии 2026-05-03 я за 47 минут не довёл ни одной задачи до конца, потому что на каждом шаге переспрашивал то, на что пользователь уже отвечал явно или в рамках. Это вывело пользователя из себя и стоило времени без выхлопа. Прямая цитата: «давишь вопросами, на которые я уже отвечал».

**How to apply:**
- Если есть сомнение — сначала проверить память (MEMORY.md, проектные файлы) и резюме предыдущих сессий, а не спрашивать заново.
- Если правда нужен ответ — задать **один** вопрос, без альтернатив на три страницы.
- В неоднозначных местах — действовать по разумному дефолту (например, типовой процессный ритм, типовой формат) и помечать «правлю по запросу».
- Лучше показать черновик и попросить «поправь, что не так» — это всегда быстрее, чем серия уточняющих вопросов.
- НЕ повторять ТЗ обратно пользователю в виде вопросов. Если ТЗ есть — делать.

**Default scope при просьбе «посмотри логи / срез / статистику» без даты — весь доступный диапазон, не последний день/час.** Кейс 2026-05-25: на «посмотри логи по нише Контент» отдал срез только за 24.05 (последний день в логе), пользователь: «не, посмотри прям за весь период а не за 24.05, прям за весь». Если пользователь хочет узкое окно — он назовёт даты. По дефолту — агрегировать всё, что есть.

**Полное исполнимое ТЗ → выполнять, не возвращать «карту/план» на подтверждение.** Кейс 2026-06-24 (Lexis Voice): пользователь дал детальное ТЗ на переписку цепочки (два списка болей, какой кейс в какое письмо, что не трогать). Я расписал в ответ «карту разводки» и спросил «подтверди карту, дальше запускаю». Пользователь: «я не буду подтверждать карту, я дал четкое тз, выполни его». Когда вход уже самодостаточен для действия — не превращать его в план-на-апрув; делать и показывать результат (его и правят). Подтверждение карты/плана уместно, только когда ТЗ реально неоднозначно и без выбора нельзя стартовать.

**Если формулировка из брифа клиента уже есть — это и есть источник, не спрашивать «можно ли её использовать».** Кейс 2026-05-27 (Мосса Инжиниринг): пользователь дал в брифе Сегмента 2 фразу «2 крупных проекта по птицефабрикам». Бренды публично не назывались, и я спросил: «можно ли упоминать описательно "2 крупных проекта по птицефабрикам" без бренда, или вообще не вставлять?». Пользователь: «не понял вопрос». Дословная цитата из брифа = подтверждённый источник, использовать как есть, описательно. Спрашивать имеет смысл только когда формулировки в брифе НЕТ и я хочу её сочинить.
