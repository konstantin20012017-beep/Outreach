Когда даю пользователю команды для запуска (scp, ssh, git, любые шелл-операции) — всегда расписывать по этапам простым языком: что эта команда делает, зачем нужна, что пользователь увидит в результате, что делать дальше.

**Why:** пользователь не разработчик в техническом смысле — голые команды без объяснения «куда я это вставляю и что произойдёт» бесполезны. Кейс 2026-05-15: дал блок scp+ssh без пояснения, пользователь попросил «объясни по-человечески».

**How to apply:** перед командами — короткое «Шаг 1: …, Шаг 2: …». После каждой команды — что должен увидеть и куда смотреть. Никаких «запускай у себя в терминале» без контекста.

**Расширение 2026-05-20 — то же касается уточняющих вопросов пользователю.** Когда нужно переспросить у пользователя что-то перед действием — формулировать ОДИН конкретный вопрос простым языком, без жаргона и составных условий. Кейс: я спросил «локальные правки уже трогали что-то новое, помимо текста стартового пинга? в git status видно M config.py / filter.py / main.py / sender.py — это всё ещё мульти-ниша рефакторинг от 19.05, или сверху ещё что-то ехало?» — пользователь сказал «простым языком задай вопрос, ничего не понял». Правильно было: «С 19.05 (когда ты последний раз деплоил) ты ещё что-то правил в коде? Да/нет.»

**Эвристика для вопроса:** если вопрос содержит технический термин (`git status`, `HEAD`, `diff`, имена файлов кодом) — переписать так, чтобы пользователь мог ответить без понимания этих терминов. Один вопрос — один ответ.
