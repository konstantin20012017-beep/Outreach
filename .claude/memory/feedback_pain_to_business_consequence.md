Каждое утверждение боли в письме надо дожимать до бизнес-следствия, понятного директору-адресату. Остановка на технической причине («установка может встать без запчастей и помощи») провисает: адресат не считывает её как СВОЮ боль.

**Why:** Сессия 2026-05-19, RefExpress цепочка C №1. Написал: «Когда на пищевом производстве расширяют или обновляют холодильный парк, рефконтейнер обычно идёт в комплекте с дизель-генератором и сервисной поддержкой. Иначе через сезон установка может встать без запчастей и помощи.» User поймал: «мне вот здесь боль непонятна, можешь пояснить?» — и был прав. Я остановился на инженерной поломке. Директор пищевого производства не думает про «без запчастей и помощи» как про боль; он думает про «склад простоит N дней → продукт портится → отгрузка сорвалась → штраф/потеря клиента». Боль обрывается на середине дуги.

**How to apply:**

- **Дуга боли:** триггер/событие → техническая причина → **бизнес-следствие для адресата**. Без третьего звена боль не считывается.
- **Шаблон «следствия» по сегментам:**
  - Пищевые/молочные/мясо/рыба: простой склада → порча сырья → срыв отгрузки → штраф/потеря клиента
  - Сертификация/документы (ECS24): просрочка → продажи остановятся → штраф/потеря тендера
  - HR/мероприятия: позже узнает → программа собрана не оптимально / места дороже / упустил спикеров
- **Самопроверка:** после каждого утверждения боли спросить «и что? что это значит в деньгах/времени для адресата?» Если не дописал — дописать.
- **Не повторять одну и ту же боль в разных цепочках на одну ЦА.** Если C №1 про «простой склада», а B №2 тоже про «простой склада» — переключить угол одной из цепочек на другую боль/сценарий.
