Если пользователь прислал путь к файлу, папке, ссылку, вложение — обработать **до** начала работы над задачей. Никогда не решать «текста в чате достаточно, файлы можно не трогать».

**Why:** в проекте Аутрич я (Claude) при работе над BPMSoft получил в одном сообщении и текст в чате, и путь к папке `#Материалы для подрядчиков` с 5 PDF. Прочитал только текст, PDF не открыл вообще. Объяснил это себе памяткой «PDF через Read не работает» — но не попробовал `pdftotext`, не сказал пользователю «вижу 5 PDF, изучить?». Молча принял решение, что текста достаточно. В итоге часть кейсов и цифр (которые есть только в PDF «Истории Успеха» и PR 2025) в цепочки не попали. Пользователь это вскрыл уже на этапе финальной правки и был прав в том, что это ломает доверие к работе.

**How to apply:**
- Первым действием при получении задачи — составить список **всех** источников (чат + файлы + папки + ссылки) и явно подтвердить «работаю со всем этим».
- Если формат файла не открывается через Read tool — попробовать альтернативу:
  - PDF: `pdftotext -layout file.pdf -` (есть в системе через homebrew)
  - DOCX: `pandoc file.docx -t plain`
  - XLSX: `python3 -c "import openpyxl; ..."` или экспорт в CSV
- Если ни одна альтернатива не работает — **явно сказать пользователю**: «такой-то файл я не могу прочесть, пришли как текст / скриншот / экспорт». Не пропускать молча.
- Памятка «PDF через Read не работает» = НЕ значит «PDF можно игнорировать». Значит «нужен другой инструмент».
- Если пользователь хочет присылать материалы поэтапно (один файл за раз с подтверждением «зафиксировал») — это его прерогатива, спросить если непонятно. Но default — он шлёт всё одним сообщением, моя задача — обработать всё.

**Веб-источники: парсить ВСЁ тело страницы, не только верхний listing.**

Кейс 2026-05-27 (Мосса Инжиниринг): я выгрузил `mossaengineering.com/projects/` через curl, прогнал grep `<h3 class="mp-c__title">` — получил 49 заголовков и принял за полный список кейсов. По цепочкам подавал кейсы под NDA-описанием. Пользователь: «я на сайте вижу 80+ кейсов. и ты хочешь сказать что не нашел нужные?». Реально на странице было **83 кейса**, причём ниже верхнего listing'а — текстовые блоки с **открытыми брендами** (МАГНИТ, РОТ ФРОНТ, ДОДО пицца, Сибирский гурман, Псковский хлебокомбинат, РУСАГРО, ФОСАГРО, Мулти Милк, КДВ групп, ах СТЕПЬ/АФК и др.), которые я мог использовать в письмах напрямую вместо «крупный производитель NDA».

**How to apply (web-парсинг):**
- При парсинге страницы сначала **проверить заявленный счётчик элементов** на странице («300+ проектов», «83 кейса на сайте», «X кейсов с отдельной страницей»). Если найденных через grep сильно меньше — listing неполный.
- Не ограничиваться одним CSS-селектором/классом. На сайтах верхний listing часто — превью, а полный набор — ниже в других блоках. Прочесть всю страницу, не выгребать одним `grep`.
- При сомнении — сделать `wc -l` файла, посмотреть структуру, прогрузить целиком в Read или прогнать второй grep по альтернативным селекторам.

**Транскрибация → бриф: не схлопывать многосоставную мысль клиента в одну строку.**

Кейс 2026-06-01 (dialeads). Триггер «вакансии» клиент на созвоне описал **двумя механиками**: A — компания набирает операционистов/менеджеров продаж = подтверждённая боль [28:38]; B (глубже, теплее) — вакансии с **AI/ИИ в стеке** у разработчиков = у компании уже есть/хочет IT-функцию, думает об автоматизации [29:28]. В бриф я записал одну строку «вакансии» — механику B потерял целиком. Всплыла, только когда пользователь попросил перечитать транскрибацию: «проанализируй еще раз всю транскрибацию… есть ли там пояснения, что имеется в виду». Это сильный, прямо проговорённый клиентом инсайт, который я выкинул при сжатии.

**How to apply (транскрибация → бриф):**
- Когда клиент проговаривает триггер/боль/УТП **в несколько слоёв или с примерами** — в бриф вынести каждый слой отдельно, с таймкодом, а не заголовок. «Вакансии» в брифе ≠ «вакансии (2 механики: A операционисты [28:38] / B AI-стек [29:28])».
- После заполнения блока брифа из созвона — пройтись по транскрибации второй раз именно по этому блоку: «что клиент сказал по этой теме, чего нет в моей формулировке?». Особенно ценны фразы «у нас сработал…», «чем хорошо…», «не только это, а ещё…» — там обычно второй слой.
- Признак, что схлопнул: в брифе строка-заголовок без расшифровки «что имеется в виду». Если на вопрос «что тут имеется в виду?» я не могу ответить цитатой из источника — значит при сжатии потерял содержание.

**SPA / hash-роутинг: curl и WebFetch отдают пустоту — нужен JS-render.**

Кейс 2026-05-29 (JEZA): сайт jeza.pro — SPA с hash-роутингом (`jeza.pro/#/cases`). Контент рисуется JS на клиенте, поэтому curl/WebFetch вернули бы пустой каркас без кейсов. Снял полный список (28 кейсов) через `firecrawl_scrape` с `waitFor: 10000` и `json`-схемой под список кейсов (title/client/category/details). Сработало с первого раза.

**How to apply (SPA):**
- Признаки SPA: URL с `#/` (hash-роут), пустой/минимальный HTML от curl, контент «появляется» только в браузере.
- Не долбить curl+grep по пустому каркасу — сразу `firecrawl_scrape` с `waitFor: 5000-10000`.
- Для конкретных данных (список кейсов, цены, поля) — `formats: ["json"]` + схема, а не markdown (точнее и сразу структурой).
- Если firecrawl с hash-URL пуст — попробовать базовый URL без `#`, либо `firecrawl_map` с `search` чтобы найти реальный путь страницы.
