После любой смысловой правки в коде проекта парсера (или другого инфра-проекта с деплоем) — **сразу** коммит с осмысленным сообщением + push на remote. Не копить недельные «незакоммиченные кучи».

**Why:** 2026-05-20 я не смог ответить на простой вопрос «есть ли свежие правки поверх последнего деплоя?» — пришлось спрашивать у пользователя. Пользователь справедливо указал: «твоя задача фиксировать все изменения, делать новые коммиты и пушить на гитхаб, поэтому это вопрос к тебе». Без коммитов после каждой итерации я теряю историю и не могу детерминированно сверить «что на сервере» vs «что локально».

**Конкретный кейс (для бдительности):** последний коммит на момент сессии 20.05 был `c825e1d` от 16.05. Между ним и сессией прошёл мульти-ниша рефакторинг 19.05 (24 файла, +2141/-227) — он уехал на VPS, но **не в git**. В результате я сначала ошибочно заявил «локальное = серверное», и баг с текстом стартового пинга всплыл только когда пользователь увидел его в TG. Если бы рефакторинг был закоммичен сразу после деплоя 19.05, я бы увидел в git status более свежие незакоммиченные правки (текст пинга) и понял, что они НЕ на сервере. Связка «коммит == граница между задеплоенным и нет» — рабочая, только если её строго соблюдать.

**How to apply:**
- Триггер: завершил смысловой блок работы (рефакторинг, новая фича, набор правок конфигов) — сразу `git add` нужного + `git commit -m "..."` + `git push`.
- В сообщении коммита — что и зачем, как делали раньше (`Этап 3: статистика, антиключи, критерии клиента, +4 чата`).
- Если делали деплой на VPS — коммит должен быть **до** или **сразу после** деплоя, не «как-нибудь потом».
- Перед началом новой сессии работы — `git status` / `git log --oneline -5`, чтобы видеть, не висит ли что-то с прошлого раза. Если висит — сначала коммитим/пушим, потом работаем.
- Перед командой пользователю «сделай рестарт сервера» — мысленно сверить: точно ли git HEAD == то, что на сервере. Если нет — сначала закоммитить разницу, чтобы было с чем сравнивать в будущем.
