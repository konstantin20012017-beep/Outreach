Когда говорю пользователю «на сервере Х», «там стоит Y», «уже задеплоено Z» — это утверждение состояния удалённой системы. Делать такие утверждения **только после фактической проверки** (ssh + команда, http-запрос, чтение файла), а не по памяти/догадке/тому-что-«должно-было-быть».

Если проверки нет — формулировать как гипотезу: «скорее всего там X, проверим», а не утверждение.

**Why:** 2026-05-20 я заявил пользователю «локальное = серверное, новых правок поверх деплоя не было — достаточно рестарта». Это было основано на git log (последний коммит 16.05) + общем впечатлении, без diff локального `main.py` с тем, что реально лежит на VPS. По факту локальный `main.py` содержал новую строку текста стартового пинга, которая на сервер не доехала. Баг всплыл только когда пользователь увидел старый текст в TG. Стоило сразу либо сделать diff (`ssh root@vps cat /opt/parser/main.py | diff - main.py`), либо честно сказать «не знаю, надо сверить».

**How to apply:**
- Триггер: я собираюсь написать в чат фразу вида «на сервере X», «уже задеплоено Y», «там стоит версия Z», «всё синхронизировано».
- Перед отправкой — спросить себя: я это **проверил в этой сессии**, или предполагаю?
- Если не проверил: либо сделать проверку (часто это одна grep/diff/curl команда), либо переформулировать как гипотезу с явным предложением проверить.
- Особенно строго — для VPS парсера, потому что прямого SSH у агента нет (см. [[feedback-parser-vps-no-password]]) → проверки идут через пользователя → каждая стоит ему действий → тем более не плодить ложные утверждения.
- Связано с [[feedback-disclose-all-silent-decisions]] (тоже про честность с пользователем) и [[feedback-commit-and-push-each-change]] (коммит как граница «задеплоено/нет»).
