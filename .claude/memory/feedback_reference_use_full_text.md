Если у проекта есть референс-кейс с реальными текстами отработавших цепочек (как iso-smk для ECS24), новая цепочка делается **наложением структуры референса блок-в-блок** на материал нового клиента, а не «авторской» сборкой на основе тезисов кейса.

**Why:** конкретный инцидент на ECS24 (2026-05-10). Я прочитал кейс iso-smk, вытащил тезисы (триггер даты, бесплатная проверка, цифры в УТП), и собрал v1 ECS24 «по правилам CLAUDE.md + выводы кейса». Структура писем получилась авторской, не наследовала референс. Пользователь сразу заметил: «структуры писем ровно такие же как у нас в ТЗ, ты точно взял подходы из примера кейса?». Пришлось переделывать v2 строго по структуре iso-smk цепочки A — это и был правильный путь с самого начала. Это та же ошибка, что была на BPMSoft (поверхностное чтение материалов), просто в другой форме: материалы прочитал, но не наложил.

**How to apply:**
- Когда у проекта есть референс с **полными текстами** успешных цепочек, обязательно открыть конкретные тексты, не ограничиваться сводкой выводов кейса.
- Перед написанием нового письма для клиента в нише с референсом — выписать структуру референса по блокам (айсбрекер → представление → буллеты → CTA → ...) и заполнить каждый блок материалом нового клиента. Менять можно: имя/компанию, продукт, цифры, 1-2 буллета регалий. Сохранять надо: порядок блоков, тип CTA в каждом письме, ритм («писала ранее», «обращаем внимание», структуру буллетов).
- Если хочется отойти от структуры референса — сначала явно сказать пользователю «здесь предлагаю отойти от референса, потому что [причина]» и получить одобрение. Не делать этого по умолчанию.
- Дополнения к референсу (например, цифры УТП клиента в блок «почему мы») — добавлять как **расширение существующего блока**, а не как замену структуры.
- Это правило важнее «креатива»: пользователь хочет получить проверенную статистикой структуру, а не мою интерпретацию.
