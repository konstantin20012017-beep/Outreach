Когда я придумал 2+ жизнеспособных варианта захода (опенер письма, угол цепочки, формулировка боли) и они **не взаимоисключающие** — дефолт не «задать вопрос, какой брать», а **сделать оба и подать как A/B-тест**. Пользователь сам отберёт по результату рассылки.

**Why:** сессия 2026-06-01 (dialeads, Сегмент 1). Я придумал два опенера письма №1 — по вакансии и по отрасли — и выкатил `AskUserQuestion` с single-select «какой делаем». Пользователь отклонил модалку и сказал: «Давай сделаем оба варианта. То есть оба варианта попробуем». Я навязал выбор там, где естественный ответ — «оба». Клиенты этого проекта A/B-тестят цепочки **по умолчанию** (ravix фитнес/ecom/b2c, exportbase реанимация/маркетплейсы — по 3 цепочки, bpmsoft 3+ цепочки, kormishin несколько форматов). Для аутрича несколько заходов на один сегмент = норма, а не развилка «или-или».

**How to apply:**
- Два разумных варианта захода, оба сорсированы, отличаются только первым абзацем/углом → собрать **оба** в одном файле как «Вариант A / Вариант Б», пометить «A/B-тест», дальше текст и CTA общие. Не спрашивать «какой выбрать».
- `AskUserQuestion` с single-select по вариантам письма — почти всегда неверный инструмент. Если вопрос правда нужен — он про **операционную развилку, которую знает только пользователь** (например «как собирается база — по вакансиям или по отрасли»), а не «какой текст тебе больше нравится».
- Single-select уместен, только если варианты реально взаимоисключающие (нельзя отправить оба одной и той же компании в одном письме) И выбор меняет всю структуру, а не один абзац.
