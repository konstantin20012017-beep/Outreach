При работе над цепочкой обязательный шаг перед написанием: `grep -niE "<ключевые термины продукта>" брифы/<проект>_исходники.md`. Бриф — это моё резюме исходников; в исходниках лежат точные формулировки клиента (определения продуктов, риски, цифры), которые в брифе могли потеряться или быть пересказаны общими словами.

**Why:** конкретный инцидент на ECS24 сегмент 3 (2026-05-10). Клиент в исходниках дал точное определение добровольного пожарного сертификата (4 строки: подтверждает соответствие нормам пожарной безопасности / по инициативе производителя или импортёра / не обязательный / для ИП и юрлиц). Я при написании №3 опирался на бриф (где этого определения не было) и iso-smk Цепочку 0, написал общими словами + добавил отсебятину про «преимущество в тендерах», «B2B-сделки» — то, чего в исходниках нет вообще. Пользователь поймал: «ты эту информацию использовал? я её в самом начале отправлял». Пришлось переписывать №3 строго по присланному определению.

**How to apply:**
- Перед написанием каждого письма цепочки: grep по `<проект>_исходники.md` на ключевые термины продукта (название продукта, синонимы, специфические слова из ниши). Например, для пожарного сертификата — `grep -niE "пожарн|противопожарн|МЧС|нормам и стандартам"`.
- Если в исходниках нашлось точное определение/формулировка/цифра — переносить **дословно или с минимальной перефразировкой**, не «своими словами по смыслу».
- Бриф использовать как навигатор и резюме структуры, но конкретные тексты для писем брать из исходников.
- В таблице «Источники» в HTML-цепочки явно указывать строку исходника, откуда взято: «<i>ecs24_исходники.md строки 42-45</i>».
- Если по какому-то блоку письма исходники молчат — это сигнал либо запросить инфу у клиента, либо написать «общими словами по согласованию» с warning-карточкой. Не закрывать пробел отсебятиной.
