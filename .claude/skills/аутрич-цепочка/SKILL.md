# Аутрич-цепочка — процессный мануал

Активируй этот skill каждый раз, когда работаешь над письмами цепочки в папке `цепочки/`. Прохождение чеклиста ниже — обязательное условие записи файла (hook блокирует Write без маркера `<!-- PREFLIGHT-OK -->`).

## ⚠️ Обязательная процедура перед Write в `цепочки/*.html`

**Перед каждым Write нового файла или существенной правкой письма** — вслух в чате пройти чеклист. Не «прочитать молча», не «учесть». Явно перечислить с отметкой ✓/✗:

1. **Source-check листом.** Все конкретные утверждения каждого письма выписать строкой, каждое пометить ✓ источник или ✗ домысел. Все ✗ — вырезать / переформулировать / в warning. Только потом Write.
2. **Распределение по письмам.** Таблица Боль/Регалии/УТП на 3 письма, без повторов между письмами.
3. **CTA-лестница.** №1 лёгкий, №2 средний, №3 либо тяжёлый, либо обнуляющий.
4. **Разводка сегментов** (если их 2+) — таблица по 4 осям: категории / триггер / боль / УТП. Не лексика, а бизнес-логика.
5. **Pre-flight grep после Write** — все 5 grep'ов возвращают пусто.

После прохождения чеклиста — вставь в HTML маркер `<!-- PREFLIGHT-OK -->` (например, сразу после `<head>`). Только тогда hook пропустит Write.

## Pre-flight по содержанию (до Write, не после)

Grep-чеклист ловит **форму**. Он НЕ ловит выдуманные смысловые утверждения. До записи цепочки:

1. **Выписать списком все конкретные утверждения каждого письма.** Каждое предложение, где есть цифра, факт, причина, перечисление, длительность, описание процесса или UX — отдельной строкой.
2. **Для каждого утверждения пометить источник:**
   - ✓ **Бриф клиента** (с указанием поля)
   - ✓ **Созвон/исходники** (с цитатой)
   - ✓ **Кейс/PDF** (с указанием раздела)
   - ✓ **Лендинг/пресс-релиз** (с URL)
   - ✗ **Мой домысел / здравый смысл** — без источника
3. **Все ✗ — обработать перед Write:**
   - либо вырезать;
   - либо переформулировать в общую фразу без конкретики;
   - либо вынести в warning-карточку и запросить у клиента;
   - НЕЛЬЗЯ оставлять как есть «потому что логично».
4. Только после этого — Write.

Это правило важнее grep-чеклиста по форме: грамматическая ошибка ловится клиентом за секунду и правится, выдуманный факт превращается в подрыв доверия и не виден на проверке без специального аудита.

## Pre-flight grep'ы после Write

Команды и полный список — в `справочники/формат-цепочек.md` раздел «Pre-flight grep'ы». Все 5 должны вернуть пусто.

## Роли писем (структура трёх писем)

**Тема (с 2026-05-29 — обязательна в каждом письме).** Отдельной строкой `Тема: ...` в начале блока, перед `[приветствие]`. **Тема ОДИНАКОВАЯ во всех 3 письмах цепочки** — чтобы письма склеивались в одну переписку в почтовике (НЕ разная под роль письма). В теме — переменная компании `{Компания}`. Тема общая по сегменту, в стиле живого менеджера, без маркетингового пафоса и капса. Давать клиенту 3-4 варианта темы на выбор. Приветствия/подписи — по-прежнему только плейсхолдеры.

У каждого из трёх писем — своя роль, не только своя боль. Это каркас, на который накладываются боли/регалии/УТП:

- **Письмо №1 — триггер + знакомство + лёгкий вопрос.** Айсбрекер под сегмент (закон/кейс/сайт/гео и т.д.) → короткое представление (кто мы, что делаем) → квалифицирующий вопрос или просьба переслать ответственному. Цель — получить любой ответ, не продавать.
- **Письмо №2 — разбор нюансов сегмента + персонализированный артефакт.** Возвращаемся к теме, расшифровываем 2–3 нюанса/боли, которые именно этот сегмент обычно недооценивает, и предлагаем готовый артефакт под получателя: КП, расчёт под его объект, визуализация с его брендингом, подборка примеров под его задачу. Чем конкретнее артефакт «под него», тем выше реплай.
- **Письмо №3 — короткий follow-up.** Одно из двух: (а) короткий «туда ли пишу — если не к вам, подскажите кому, буду благодарен» (2–3 строки, без оффера), либо (б) квалифицирующий вопрос на горизонт «если в ближайшее время планируется проект — напишите, иначе больше не беспокою». Не повторять оффер из №2, не давить.

**PS «пишу на несколько почт» (опционально, для письма №2 или №3)** — если рассылаем по нескольким адресам компании: «PS – пишу на несколько почт, если попал не по адресу — перешлите кому надо, буду благодарен)». Снимает неловкость и открывает форвард внутри компании.

## CTA-лестница

Шаг должен расти от письма к письму, не наоборот:
- **№1** — самый лёгкий: квалифицирующий вопрос, «кому переслать», «прислать пример/кейс?», «ответить да/интересно». Никаких созвонов и расчётов.
- **№2** — средний: готовый артефакт под получателя (КП, расчёт, визуализация, подборка), либо просьба дать вводные для расчёта.
- **№3** — тяжёлый или обнуляющий: предметный созвон/встреча/выезд, либо короткий follow-up «туда ли пишу».

**Варианты CTA:** номер телефона (по умолчанию), презентация, лид-магнит, ответ «да», **примеры работ** («могу прислать примеры — посмотрите за 5 минут и сами решите»). Примеры работ — сильный CTA для ниш где качество оценивается визуально, ставить наравне со скидкой или впереди. Скидку подавать как «дарим», а не «даём». Длинный цикл сделки = более лёгкий CTA. Цель — перевести из почты в звонок/мессенджер.

## Распределение по письмам (без дубликатов)

- **В каждом письме — своя ключевая боль.** Распределить разные боли из брифа по №1, №2, №3. Боли разные по сути, не переформулированные («не хватает скорости» и «перенастройка съедает время» — это про одно).
- **Регалии не повторять между письмами.** Варианты: опыт (лет), бренды-клиенты, экспертиза в нише, работа по KPI, 80% клиентов работают 1+ год, не ведём конкурентов в одном гео.
- **Таблица распределения** в HTML перед блоком «Источники»: строки — письма (№1, №2, №3), колонки — Боль / Регалии / УТП. Визуальный контроль на дубликаты.

## Разводка сегментов одного клиента

- **Уникальные сегменты — пишутся максимально по-новому.** НЕ копировать формулировки из других цепочек этого же клиента. Менять: айсбрекер, заход к боли, главный УТП-аргумент, переходы между блоками.
- **Что должно отличаться:** первая фраза, формулировка перехода («самое частое» / «частая жалоба» / «частая история»), сами боли, главные УТП, цифры, кейсы.
- **Что можно оставлять неуникальным:** «Возвращаюсь к прошлому письму» во 2-м, «Напомню о себе коротко» в 3-м, базовая структура айсбрекер→боль→решение→CTA, плейсхолдеры `[приветствие]` / `[подпись]`, регалии завода (но обвязка разная).
- **Применимо и к разным клиентам в одной нише** — для агентства Б в нише Х фразы и заходы отличаются от агентства А.

### Один продукт — N сегментов: разное «почему именно вам»

Если один продукт продаётся в несколько сегментов, в письме №1 каждого сегмента — своя бизнес-логика, не переписанный айсбрекер. Пример (зарядные станции):
- застройщикам — «покупатели электрокаров платят за инфраструктуру, продадите квартиры дороже»;
- АЗС — «с 1 марта обязаны установить, иначе нарушение»;
- ТЦ — «водитель заряжается 30 минут и оставляет деньги в фудкорте»;
- отели — «гость дольше остаётся, выше средний чек»;
- УК — «у жильцов уже есть электрокары».

В брифе под каждый сегмент фиксировать строку «почему именно вам» — она становится ядром письма №1.

## Кейсы и регалии

- В разных письмах одной цепочки — **разные кейсы**, не повторять названия и цифры.
- **Авторство:** «Вот что мы сделали для X», не «Вот что получилось у X».
- **Если у клиента нет конкретных кейсов с цифрами** — НЕ выдумывать. Использовать регалии (опыт лет, число клиентов, общие цифры). В HTML — warning-карточка «конкретных кейсов нет, обновим до vN после получения». Запрашивать у клиента.
- **Крупные клиенты при первом знакомстве** — упоминать узнаваемые бренды портфолио (Газпром, Кофемания, Мать и Дитя). Локальные ООО для холодного получателя имени не дают.
- **Соцпруф клиентами — всегда именами, без обобщений.** Запрещённые формулировки: «среди заказчиков есть крупные компании», «работают крупные игроки рынка», «нам доверяют большие бренды». Это пустой звук, холодный получатель проверяет глазами. Заменять на конкретные публичные бренды из портфолио («Среди заказчиков — VK, СБЕР ЗВУК, BACARDI»). Условия: (1) только публичные кейсы (сайт/пресс), NDA не называть; (2) разносить бренды по письмам цепочки, не повторять одни и те же; (3) если узнаваемых публичных имён нет — это open field в warning, запросить у клиента, обобщённую фразу НЕ оставлять.
- **Размытые формулировки убирать:** «похожая ситуация» (с чем?), «насколько это вам подходит» (что подходит?), «решаем задачи разного уровня» — заменять конкретикой (что именно, для кого, с каким результатом).

## Не выдумывать конкретику продукта

Распространяется на **любую** конкретику продукта/услуги, которой нет в брифе:
- **Цифры эффективности** («закрываем 80% вакансий», «срок 14 дней», «удержание 90%») — если не дал клиент, НЕ писать.
- **Модус работы услуги** (как агентство ищет, сравнивает, ведёт замену) — НЕ додумывать.
- **Критерии отбора, методология, фишки процесса** — только от клиента.
- **Расширение списка ролей/услуг** за пределы брифа. «Сборщик, контролер ОТК, инженер» — не дописывать «бригадиры, технологи, мастера».
- **Причины оттока клиентов** — «самое частое с чем спотыкались — X» — не писать без статистики.
- **Варианты ответов получателя в его уста** — «база не подошла по качеству / фильтрам / цене» превращается в утверждение «по этим причинам обычно уходят».
- **Способы работы получателя** — «через сайты, выписки, парсинг, подписки» — мой перечень. Задавать вопрос без перечня.
- **Этапы процессов получателя** — «в ручном сборе уходит время на проверку email» — гипотезы. Не писать без источника.
- **Длительность встреч/звонков** («15 минут на разбор», «полчаса на демо») — если не сказано, не зашивать число. «Короткий звонок» — норм.
- **UX/функционал кабинета или сайта** — «там можно собрать выборку», «в кабинете есть фильтр X» — НЕ писать без подтверждения. Получатель проверит — доверие сгорит.
- **Допущения про админку клиента** («Вижу по кабинету, что вы пробовали», «По нашим логам вы заходили N раз») — не приписывать видимость.

**Допустимо:** общие утверждения из брифа («подбираем под роль», «работаем с производствами 300-600 млн»), регалии головного бренда, плейсхолдеры подниш `[вариант 1 / вариант 2]`.

Если без конкретики письмо проседает — не закрывать пробел фантазией. Открытое поле в warning-карточку, запросить у клиента.

## Плейсхолдеры подниш в письме №1

- Если сегмент покрывает несколько подниш, и у каждого получателя своя — формат `[вариант 1 / вариант 2 / вариант 3]` в квадратных скобках через слеш.
- Оператор перед отправкой выбирает один вариант.
- Все варианты — реальные подниши из брифа (не выдуманные).

## Регламентный дедлайн ≠ тайм-оффер

Если на лендинге реальный регламентный дедлайн (анкета СБ к дате X, лимит мест по регламенту, дата закрытия регистрации) — можно использовать в дожиме как **естественный крючок срочности**:
- «Регистрация закрывается 4 июня — после этой даты допуск по регламенту не оформят».
- Подаётся как факт регламента, а не «успейте до...».
- Не оферт, не скидка, не тайм-оффер. Юридика чистая.

Запрещено маскировать под регламент придуманную срочность («осталось 3 места», «цена вырастет после X») если этого нет в лендинге.

## Кросс-сейл смежных событий в дожиме

Может быть полезен в письме №2 — упоминание ближайших мероприятий через двоеточие («останется контакт: 3 июня — Сима-ленд, 11-13 июня — Аскона»). Но:
- **Совпадение ЦА обязательно.** Между событиями для одной аудитории (миссии 500+ млн). Между разными ЦА (миссии 500+ млн и HR-продукт 300-600 млн) — НЕ ставить.
- **Один CTA на письмо.** Кросс-сейл = опциональный «остаётся контакт», не альтернативный CTA.
- **Только реальные ближайшие события** из календаря клиента, без выдумывания.

## Услуговые/постоянные продукты ≠ мероприятия

Если у клиента продукт-услуга с постоянным потоком заявок (HR-агентство, аутсорс, абонентка) вместо разового мероприятия:
- Структура «Приглашаю → Дата → Место → Программа → CTA» **не работает** — нет даты и места.
- Использовать **квалифицирующий вопрос** как заход письма №1: «У вас сейчас открыта вакансия [роль]?» / «Есть задача по X?».
- Тон **информационный**: «пишу от агентства / запустили направление», без «я знаю вашу проблему».
- Регалии — головного бренда, не нового продукта (если у продукта нет своего трекрекорда).
- Письмо №2 — дожим через вторую боль + повтор CTA. Без кросс-сейла на смежные мероприятия если ЦА не совпадает.

## A/B-тест цепочками на одной нише

Если у проекта **нет отраслевой/сегментной привязки** (ЦА — все средние/крупные компании от X выручки, без разделения по отраслям) и пользователь просит проверить какой заход лучше — делаем не сегменты, а **N цепочек на одну ЦА с принципиально разными заходами**:
- Цепочка №1 — один угол (например, по референсу-кейсу).
- Цепочка №2 — другой (бренды/регалии).
- Цепочка №3 — третий (конкретная боль / миграция / ROI).

Внутри каждой — стандарт проекта (3 письма, разные боли по письмам). Между цепочками **боли и заходы не пересекаются**.

При наличии референса от пользователя (готовая цепочка из чужого кейса) — одна копирует логику референса 1:1, остальные идут по правилам проекта.

## Работа с базой

Технические вопросы запуска рассылок (домены, прогрев, ESP) — не зона этого skill'а. Но если клиент в брифе/созвоне даёт данные по базе — учитывать:
- **Фильтры по выручке/гео/ОКВЭД** под сегмент — отразить в брифе.
- **Приоритизация почт после первой волны** — клиент анализирует ответы (какие домены/форматы лучше отвечают), это влияет на v2: пересобрать айсбрекер под профиль реально отвечающих.

Если в брифе таких данных нет — не выдумываем.

## Референс — накладывать блок-в-блок

Если у клиента есть полные тексты отработавших цепочек (например, iso-smk кейс) — копируем структуру и подставляем материал клиента. Не «по выводам кейса», не пересказ тезисами.

## Изучить материалы — команды

Правило «изучить ВСЕ материалы ДО работы» — красная линия в CLAUDE.md. Конкретные команды для нечитаемых форматов:

- **PDF:** `pdftotext -layout file.pdf -`
- **DOCX:** `pandoc file.docx -t plain`
- **XLSX:** `python3 -c "import openpyxl; wb=openpyxl.load_workbook('file.xlsx'); [print(r) for r in wb.active.iter_rows(values_only=True)]"` или экспорт в CSV

## Перед цепочкой grep'нуть исходники

Конкретные определения и формулировки клиента лежат в `*_исходники.md`, в брифе только резюме. Переносить дословно.

## Согласование рода глаголов с подписью

Если подпись женская, во всех 3 письмах «увидела/нашла/благодарна». Grep'ать перед сдачей.

## Не использовать канцелярит

«Подсказать по выгоде», «оптимизировать процесс», «реализовать возможность» — не писать. Читать вслух, переписывать всё чего не сказал бы живой человек в переписке.

## Присланные примеры — не правила формата

Из примеров клиента берём только **содержание** (факты, спикеры, цифры, тон). Формат (число писем, тема, приветствие, переменные) всегда из CLAUDE.md/skill'а.

**Исключение:** если клиент **явно просит** нестандартный формат (например, «по такому же как 02.04») — следуем просьбе и фиксируем исключение в брифе сегмента, не в CLAUDE.md.
