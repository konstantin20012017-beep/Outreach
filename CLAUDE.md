# Outreach — Email-цепочки и рассылки

Рабочая папка для создания email-цепочек и писем для холодного аутрича.

## Структура

- `брифы/` — брифы, кейсы, исходники по каждому проекту/сегменту
- `цепочки/` — готовые цепочки в HTML
- `справочники/формат-цепочек.md` — типографика, HTML-формат, кнопки копирования
- `.claude/skills/аутрич-стратегия/` — стратегические принципы (сегментация, триггеры, формулы письма, мимикрия)
- `.claude/skills/аутрич-цепочка/` — процесс создания цепочки (pre-flight, роли писем, CTA-лестница, разводка сегментов, кейсы, плейсхолдеры, кросс-сейл)

Skill'ы подгружаются автоматически по триггерам. Если по какой-то причине не сработали — прочитать вручную перед работой.

## 🔴 Красные линии — нарушение = переделать с нуля

### 1. Не выдумывать конкретику клиента

Цифры эффективности, кейсы, модус работы услуги, критерии отбора, расширение списков ролей/услуг за пределы брифа, причины оттока, варианты ответов получателя, способы работы получателя, этапы внутренних процессов, длительность встреч, UX кабинета/сайта, допущения про админку — **ничего из этого не писать**, если нет источника в брифе/исходниках/кейсе/лендинге.

Если без конкретики письмо проседает — открытое поле в warning-карточку, запросить у клиента. Не «логично домыслить».

Подробный список с примерами и допустимыми исключениями — в skill `аутрич-цепочка`.

### 2. Каждая цифра — с источником

Любая конкретная цифра (% эффективности, кратность, срок, сумма) атрибутирована к источнику. Форма подачи зависит от типа источника (кейс компании / общий бриф / только лендинг) — детали в skill `аутрич-стратегия` раздел «Каждая цифра — с привязкой к источнику».

В HTML под цепочкой — блок «Источники» с точной ссылкой на каждую цифру.

### 3. Не писать темы, приветствия, подписи

Только плейсхолдеры `[приветствие]` и `[подпись]` в квадратных скобках. Темы письма не писать вообще.

Из присланных клиентом примеров берём только **содержание** (факты, цифры, тон), формат — всегда из этого CLAUDE.md / справочника / skill'а.

**Исключение:** прямая просьба клиента (формат отработавших ранее цепочек) — следуем просьбе, фиксируем как исключение в брифе сегмента.

### 4. Никаких антитез «не X, а Y»

Запрещены конструкции противопоставления: «не X, а Y», «X, а не Y», «вместо X — Y». Только прямое утверждение того, что есть.

Антитеза читается как реклама и самооправдание, ломает мимикрию под живого менеджера.

Обычные отрицания («без раздувания штата») — норма. Запрещены именно противопоставления.

### 5. Любая входящая информация → сразу в бриф

Как только пользователь присылает по проекту **транскрибацию созвона, голосовое, описание ниши, ссылки на сайт, кейсы, скриншоты, выдержки из переписки** — немедленно создать или обновить `брифы/<проект>_бриф.md` (и `.html`) и зафиксировать ключевое: ниша, ЦА/сегменты, продукт, боли, УТП, регалии, кейсы с цифрами, CTA, контакт, открытые вопросы.

- **До** обсуждения цепочек, не «потом».
- Если проект новый и брифа нет — создать первым же действием.
- Если бриф есть — дописать блок с датой («## Дополнение от YYYY-MM-DD»).
- В MEMORY.md — пункт о проекте со ссылкой на бриф.
- Открыть `.html` бриф в Google Chrome сразу.
- Сырые материалы (транскрибация, голосовые) сохранять рядом: `брифы/<проект>_исходники.md`.

### 6. Изучить ВСЕ присланные материалы ДО начала работы

Если пользователь присылает путь к файлу/папке/ссылке — обработать **до** того как писать клиентский текст. Никогда не решать самостоятельно «текста в чате достаточно».

- Первым действием — перечислить все источники (чат + файлы + папки + ссылки) и подтвердить «работаю со всем этим».
- Нечитаемые форматы — через альтернативные инструменты (pdftotext / pandoc / openpyxl — конкретные команды в skill `аутрич-цепочка` раздел «Изучить материалы»).
- Если ни одна альтернатива не работает — **явно сказать пользователю**: «такой-то файл прочесть не могу, пришли как текст/скриншот/экспорт». Не пропускать молча.
- «PDF через Read не работает» ≠ «PDF можно игнорировать». Значит «нужен другой инструмент».

### 7. Перед Write в `цепочки/*.html` — PRE-FLIGHT блок и маркер

**Hook блокирует Write без маркера `<!-- PREFLIGHT-OK -->` в файле.** Маркер вставляется ТОЛЬКО после того, как в чате выведен блок PRE-FLIGHT с:

1. **Source-check листом** — все конкретные утверждения каждого письма списком, каждое ✓ источник или ✗ домысел. Все ✗ — вырезать / переформулировать / в warning.
2. **Таблицей распределения** по письмам — Боль/Регалии/УТП на 3 письма, без повторов.
3. **CTA-лестницей** — №1 лёгкий, №2 средний, №3 тяжёлый/обнуляющий.
4. **Разводкой сегментов** (если их 2+) — по 4 осям: категории / триггер / боль / УТП. Не лексика, а бизнес-логика.

После Write — 5 grep'ов из справочника `формат-цепочек.md` должны вернуть пусто.

### 8. Без «ё», тире `–` для текста, дефис `-` для диапазонов цифр

- «ё» — никогда. Всегда «е».
- Тире `–` (короткое) для текста и перечислений. Длинное `—` запрещено.
- Дефис `-` для диапазонов цифр без пробелов: «2-5 минут», «1-3 мм». Не через тире.

## Чек-лист идеальной цепочки

Полный чеклист — `чеклист_цепочки.md` (если файл существует) или раздел «Обязательная процедура» в skill `аутрич-цепочка`. Проходить вслух перед каждой записью в `цепочки/*.html`, не «прочитать молча». Пользователь имеет право остановить и потребовать переделать, если процедуры в чате не было.
