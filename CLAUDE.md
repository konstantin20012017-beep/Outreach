# Outreach — Email-цепочки и рассылки

## Что это за проект

Рабочая папка для создания email-цепочек и писем для холодного аутрича. Все материалы, цепочки и шаблоны хранятся здесь.

## ОБЯЗАТЕЛЬНАЯ ПРОЦЕДУРА перед записью в `цепочки/*.html`

**Перед каждым `Write` нового файла или существенным правкам письма** — вслух в чате пройтись по `чеклист_цепочки.md`. Не «прочитать молча», не «учесть». Явно перечислить ключевые пункты с отметкой ✓/✗, чтобы пользователю было видно, что процедура выполнена:

1. **Source-check листом** — все конкретные утверждения каждого письма выписать строкой, каждое пометить ✓ источник или ✗ домысел. Все ✗ — вырезать / переформулировать / в warning. Только потом Write.
2. **Распределение по письмам** — таблица Боль/Регалии/УТП на 3 письма, без повторов между письмами.
3. **CTA-лестница** — №1 лёгкий, №2 средний, №3 либо тяжёлый, либо обнуляющий.
4. **Pre-flight grep после Write** — все 5 grep'ов из раздела «Pre-flight чеклист» возвращают пусто.

Если процедуры в чате не было — пользователь имеет полное право остановить и потребовать переделать. Это договорённость, а не предложение.

## Правило №0: Любая входящая информация по проекту → сразу в бриф

Как только пользователь присылает по проекту **транскрибацию созвона, голосовое, описание ниши, ссылки на сайт, кейсы, скриншоты, выдержки из переписки** — немедленно создать или обновить файл `брифы/<проект>_бриф.md` (и `.html`) и зафиксировать туда всё ключевое: ниша, ЦА/сегменты, продукт, боли, УТП, регалии, кейсы с цифрами, CTA, контакт, открытые вопросы.

- Делать это **до** обсуждения цепочек, не «потом» — иначе данные теряются между сессиями.
- Если проект новый и брифа ещё нет — создать его первым же действием.
- Если бриф уже есть — дописать новый блок с датой («## Дополнение от YYYY-MM-DD»), а не перезаписывать.
- В MEMORY.md добавить пункт о проекте со ссылкой на бриф.
- Открыть `.html` бриф в Google Chrome сразу после создания/обновления.

Сырые материалы (транскрибация, голосовые расшифровки) сохранять рядом: `брифы/<проект>_исходники.md`.

## Правило №0.5: Изучить ВСЕ присланные материалы ДО начала работы

Если пользователь присылает путь к файлу/папке/ссылку — обработать **до** того, как писать что-либо клиентское. Никогда не решать самостоятельно «текста в чате достаточно, файлы можно не трогать».

- Первым действием — перечислить все источники (чат + файлы + папки + ссылки) и подтвердить «работаю со всем этим».
- Если формат не открывается через Read tool — пробовать альтернативу:
  - **PDF:** `pdftotext -layout file.pdf -` (есть через homebrew, работает в этой системе)
  - **DOCX:** `pandoc file.docx -t plain`
  - **XLSX:** `python3 -c "import openpyxl..."` или экспорт в CSV
- Если ни одна альтернатива не работает — **явно сказать пользователю**: «такой-то файл прочесть не могу, пришли как текст/скриншот/экспорт». Не пропускать молча.
- Памятка «PDF через Read не работает» = НЕ значит «PDF можно игнорировать». Значит «нужен другой инструмент».

## Правило №0.6: Каждая цифра в письме — с привязкой к источнику

Когда в письме появляется конкретная цифра (% эффективности, кратность, срок, сумма) — она обязана быть атрибутирована, и атрибуция определяет форму подачи в письме:

- **Если цифра есть в кейсе (PDF «Истории Успеха», пресс-релиз, видеокейс)** — атрибутировать к компании в самом тексте: «Билайн — средний чек по сделкам +24%», «АСКОНА — конверсия онлайн-продаж x1,5», «ВОЛМА — закупки -20%». Это самая сильная форма.
- **Если цифра только в общем брифе клиента (свод метрик «на основе реальных проектов» без привязки к одному кейсу)** — давать как «в среднем по проектам BPMSoft / агентства / завода»: «маркетинговые расходы – минус 50%». Не приписывать конкретной компании, которой эта цифра в кейсе не подтверждена.
- **Если цифра звучит только на лендинге мероприятия / на сайте, но в кейсах её нет** — подавать как речь спикера / тезис вебинара: «спикер расскажет, как ускорил БП в 5 раз», «на вебинаре разберём, как повысили эффективность на 30%». Не утверждать как наш факт.

В блоке «Источники» в HTML-цепочки каждая цифра должна иметь точную ссылку: PDF + раздел / пресс-релиз + дата / лендинг + URL / транскрибация + таймкод.

## Стратегия аутрича (ключевые принципы)

### 1. Узкая сегментация
- Не широкие базы, а конкретные сегменты (например: стоматологии, автосервисы и т.д.)
- Релевантность — главный фактор

### 2. Персонализация через ценность
- В письмо закладываются 2–3 конкретные идеи под клиента (на основе данных: сайт, описание, гео + AI)
- Каждый получатель видит релевантные именно ему предложения

### 3. Структура письма: Айсбрекер → Проблема → Решение → CTA
Три формулы (выбирать в зависимости от ниши и наличия данных):
- **Боль → CTA** — базовая формула
- **Боль + Доказательство** (цифры, кейс) → CTA
- **Боль + Частичное решение / Инсайт** (подсказка, от которой клиент задумается) → CTA

### 4. Айсбрекер на основе триггера (сигнала)
Письмо должно выглядеть так, будто менеджер лично изучил компанию. Типы триггеров:
- **Наличие кейса** — «увидели, что вы в сфере X, у нас есть кейс с вашими коллегами по рынку»
- **Описание компании / сайт** — «побывал на сайте, увидел, что у вас есть [конкретная услуга/продукт]»
- **Гео** — «мы тоже из [город]»
- **Маркетплейсы** — «увидели вас на маркетплейсах»
- **Вакансии** — как сигнал потребности
- **События / новости** компании
- **Технологический стек** (CMS, инструменты аналитики)
- **Закон / регуляторный дедлайн** — самый сильный заход, если в нише реально есть такая боль/обязательство (например, «с 1 марта обязали ставить X», «требования идут в регионы», субсидия от государства). НЕ применять, если регуляторной боли в нише нет — не пугать клиентов выдуманными «обязательствами». Использовать только когда в брифе зафиксирован реальный закон/норматив, бьющий по получателю. Если триггер есть — конкретные цифры регуляторики (даты, проценты, мощности, размер субсидии) можно давать прямо в первом письме, не дожидаясь второго.

### 5. Если нет явного триггера — сначала квалифицирующий вопрос
- Не утверждать, а спросить: «У вас есть X?» / «Планируете ли Y?»
- Потом объяснить: «Почему спрашиваю — дело в том, что...» и дать краткий оффер с цифрами
- Это снижает негатив от нерелевантных писем и сразу отсеивает нецелевых
- **В последнем письме цепочки** — квалифицирующий вопрос особенно уместен: «Если сейчас есть проект в работе / планируется...». Фильтрует тех, кому сейчас не актуально, и снижает давление

### 6. Мимикрия под живого менеджера
- Письмо НЕ должно выглядеть как рассылка
- В некоторых нишах — чем проще и «деревяннее», тем лучше
- Подстраиваться под аудиторию: для «гаражников» — максимально просто, для IT — можно сложнее
- **Для новых компаний** — информационный тон: «мы запустили / пишу от студии» вместо «я знаю вашу проблему». Задача — донести, что появились на рынке, а не давить на боли
- **PS-блок «это не AI»** (опционально, для первого письма) — приём для ниш, где аудитория привыкла получать AI-рассылки (IT, digital, маркетинг). PS в конце первого письма снимает подозрение: подчеркнуть, что письмо написано лично и отправитель реально изучил компанию. Пример: «PS – Понимаю, что сейчас много автоматических AI-рассылок. Это письмо я написал лично и действительно посмотрел ваши работы.»

### 7. CTA — мягкий, соответствующий циклу сделки
- Не закрывать сразу на КП / планировку / расчёт — слишком тяжёлое действие для холодного контакта
- Варианты CTA (указываются в брифе): **номер телефона** (по умолчанию), презентация, лид-магнит, ответ «да», **примеры работ** («могу прислать примеры — посмотрите за 5 минут и сами решите»)
- **Примеры работ** — сильный CTA для ниш, где качество оценивается визуально. Ставить наравне со скидкой или впереди неё
- Скидку подавать как «дарим», а не «даём» — мягче позиция, не выглядит как нужда
- Длинный цикл сделки = более лёгкий CTA
- Цель — перевести общение из почты в звонок/мессенджер (продолжать в почте = потеря конверсии)
- **Лестница CTA по письмам цепочки** — шаг должен расти от письма к письму, не наоборот:
  - Письмо №1 — самый лёгкий шаг: квалифицирующий вопрос, «кому переслать», «прислать пример/кейс?», просто ответить «да/интересно». Никаких созвонов и расчётов в первом письме.
  - Письмо №2 — средний шаг: готовый артефакт под получателя (КП, расчёт, визуализация, подборка примеров под его задачу), либо просьба дать вводные для расчёта.
  - Письмо №3 — самый тяжёлый или, наоборот, обнуляющий: либо предметный созвон/встреча/выезд, либо короткий follow-up «туда ли пишу — если нет, подскажите кому» (см. раздел «Роли писем в цепочке»).

### 7.1. Никаких антитез в тексте
- Запрещены конструкции противопоставления: «не X, а Y», «X, а не Y», «вместо X — Y», «не расскажем, как любим людей, а конкретный разбор» и любые их варианты.
- Только прямое утверждение того, что есть: «конкретный разбор найма и продаж iSpring» (без отсылки к «не маркетинговой презентации»).
- Антитеза читается как реклама и самооправдание, ломает мимикрию под живого менеджера.
- Перед сдачей цепочки делать grep `не .*, а |, а не ` и переписывать найденное.
- Обычные отрицания («без раздувания штата») — норма, запрещены именно противопоставления двух вариантов.

### 8. Язык аудитории (для твёрдых ниш)
- Писать на языке клиента: использовать фразы, с которыми клиенты обращаются впервые
- Использовать точные формулировки из звонков и брифов
- Включать возражения из отдела продаж
- Конкретика: цифры, бренды, факты
- Чем короче, твёрже и понятнее — тем лучше
- «Мы местные, тут работаем» — если применимо

## Формат писем

- **Тему письма НЕ писать** — только содержание
- **Приветствие и подпись** — писать именно `[приветствие]` и `[подпись]` в квадратных скобках, не писать текст приветствия/подписи (они вставляются отдельно)
- Письма должны быть короткими и конкретными
- **Буква «ё»** — никогда не использовать. Всегда писать «е» вместо «ё» (например: «еще», «все», «ведем»)
- **Тире** — всегда использовать «–» (короткое тире), а не «—» (длинное тире). Это касается и перечислений через тире в тексте письма (например: «– работаем в вашем стиле»)
- **Числовые диапазоны** — через дефис-минус «-» без пробелов, не через тире «–». Правильно: «2-5 минут», «1-3 мм», «6-8 месяцев», «10 000-40 000 метров». Неправильно: «2–5 минут», «1–3 мм». То есть: для текста и перечислений — тире «–», для диапазонов цифр — дефис «-»
- **Сохранение цепочек** — всегда сохранять готовые цепочки в папку проекта
- **HTML-формат** — всегда выводить цепочки в HTML-файл с кнопкой копирования для каждого письма отдельно + кнопка «Копировать все письма» в начале
- **3 письма в цепочке** — всегда ровно 3 письма, не больше и не меньше
- **Нумерация писем** — всегда писать «Письмо №1», «Письмо №2», «Письмо №3» (не капсом, именно с заглавной буквы)
- **Межстрочный интервал** — всегда 1.15 (line-height: 1.15) в HTML
- **Шрифт** — всегда Arial, размер 11pt для основного текста
- **Заголовки** — названия сегментов («Сегмент №1 — фитнес-клубы») и номера писем («Письмо №1») оформлять как Заголовок 2 Google Документов: Arial 18pt, жирный (`<h2 style="font-family: Arial; font-size: 18pt; font-weight: bold;">`). Это нужно, чтобы при копировании из HTML в Google Документы форматирование подхватилось автоматически (заголовки распознавались как Heading 2, тело как обычный текст Arial 11pt). Все элементы письма должны иметь явный inline-style с font-family: Arial и font-size: 11pt — без этого Google Документы при вставке могут поломать шрифт
- **Источники** — перед цепочкой указывать точные цитаты из брифа/кейсов/созвона, на которых основан каждый элемент письма
- **Открывать готовые файлы** — всегда открывать готовые HTML-файлы (цепочки и брифы) в Google Chrome (`open -a "Google Chrome"`) сразу после создания — откроется новой вкладкой в текущем окне
- **Брифы тоже в HTML** — брифы создавать в HTML-формате (не markdown), чтобы читались в браузере. Markdown-версию можно оставлять как исходник

## Роли писем в цепочке (структура трёх писем)

У каждого из трёх писем — своя роль, не только своя боль. Это каркас, на который накладываются боли/регалии/УТП:

- **Письмо №1 — триггер + знакомство + лёгкий вопрос.** Айсбрекер под сегмент (закон/кейс/сайт/гео и т.д.) → короткое представление (кто мы, что делаем) → квалифицирующий вопрос или просьба переслать ответственному. Цель — получить любой ответ, не продавать.
- **Письмо №2 — разбор нюансов сегмента + персонализированный артефакт.** Возвращаемся к теме, расшифровываем 2–3 нюанса/боли, которые именно этот сегмент обычно недооценивает, и предлагаем готовый артефакт под получателя: КП, расчёт под его объект, визуализация с его брендингом, подборка примеров под его задачу. Чем конкретнее артефакт «под него», тем выше реплай.
- **Письмо №3 — короткий follow-up.** Одно из двух: (а) короткий «туда ли пишу — если не к вам, подскажите кому, буду благодарен» (2–3 строки, без оффера), либо (б) квалифицирующий вопрос на горизонт «если в ближайшее время планируется проект — напишите, иначе больше не беспокою». Не повторять оффер из №2, не давить.

**PS «пишу на несколько почт» (опционально, для письма №2 или №3)** — если рассылаем по нескольким адресам компании, в конце уместно добавить: «PS – пишу на несколько почт, если попал не по адресу — перешлите кому надо, буду благодарен)». Снимает неловкость и одновременно открывает форвард внутри компании.

## A/B-тест цепочками на одной нише (когда сегментировать нельзя)

Если у проекта **нет отраслевой/сегментной привязки** (ЦА — все средние/крупные компании от X выручки, без разделения по отраслям) и пользователь просит проверить, какой заход лучше работает, — делаем не сегменты, а **N цепочек на одну и ту же ЦА с принципиально разными заходами**:

- Цепочка №1 — один угол захода (например, по референсу-кейсу, который дал пользователь).
- Цепочка №2 — другой угол (бренды/регалии).
- Цепочка №3 — третий угол (конкретная боль / миграция / ROI).

Внутри каждой цепочки — стандарт проекта (3 письма, разные боли по письмам, [приветствие]/[подпись] и т.д.). Между цепочками — **боли и заходы не пересекаются** (это смысл теста).

При наличии референса от пользователя (готовая цепочка из чужого кейса) — одна из цепочек копирует логику референса 1:1, остальные идут по правилам проекта.

## Один продукт — N сегментов: разное «почему именно вам»

Если один и тот же продукт продаётся в несколько сегментов, в первом письме каждого сегмента должна быть **своя причина-мотивация под боль сегмента**, а не просто переписанный айсбрекер. Пример (зарядные станции, один продукт):
- застройщикам — «покупатели электрокаров платят за инфраструктуру, продадите квартиры дороже»;
- АЗС — «с 1 марта обязаны установить, иначе нарушение»;
- торговым центрам — «водитель заряжается 30 минут и за это время оставляет деньги в фудкорте»;
- отелям — «гость дольше остаётся на территории, выше средний чек»;
- УК — «у жильцов уже есть электрокары и они выносят мозг с установкой».

Это глубже, чем «менять формулировки между сегментами»: меняется сама **бизнес-логика «почему вам это выгодно»**, а не только лексика. В брифе под каждый сегмент фиксировать отдельную строку «почему именно вам» — она и становится ядром письма №1.

## Работа с базой (если данные есть в брифе)

Технические вопросы запуска рассылок (домены, прогрев, ESP) — не зона CLAUDE.md. Но если клиент в брифе/на созвоне даёт информацию по базе — фиксировать и учитывать при проектировании цепочки:
- **Фильтры по выручке/гео/ОКВЭД** под сегмент (отсекать тех, кому продукт не по карману или до кого не доехать) — если есть данные, отразить в брифе сегмента.
- **Приоритизация почт после первой волны** — если клиент будет анализировать ответы первой волны (какие домены/форматы адресов лучше отвечают), это влияет на v2 цепочки: можно пересобрать айсбрекер под профиль реально отвечающих.

Если в брифе таких данных нет — не выдумываем, ограничиваемся сегментацией по болям и продукту.

## Pre-flight чеклист перед отдачей цепочки

Прогнать grep по HTML-цепочки и переписать всё, что нашлось:

- `grep -nE "Тема:|Здравствуйте|Добрый день|доброгоВремениСуток|С уважением|^Подпись" file.html` — темы, приветствия, подписи в теле письма (должно быть пусто, кроме `[приветствие]` / `[подпись]`).
- `grep -nE "не [А-Яа-я]+.*, а |, а не |вместо .* —" file.html` — антитеза «не X, а Y» / «вместо X — Y» (запрещена).
- `grep -nE "ё|Ё" file.html` — буква ё (всегда заменять на «е»).
- `grep -nE "[0-9]–[0-9]" file.html` — диапазоны цифр через тире вместо дефиса (должен быть «-»).
- `grep -nE "—" file.html` — длинное тире (использовать только короткое «–»).

Присланные клиентом примеры — не правила формата. Из них берём только содержание (факты, спикеры, цифры, тон), формат (число писем, тема, приветствие, переменные) всегда из этого CLAUDE.md.

## Структура папок

- `брифы/` — брифы и кейсы по каждому проекту/сегменту
- `цепочки/` — готовые цепочки в HTML-формате

## Правила работы с кейсами

- В разных письмах одной цепочки всегда использовать **разные кейсы** — не повторять названия и цифры
- **Авторство кейса** — всегда ясно указывать, что это работа агентства: «Вот что мы сделали для X», а не «Вот что получилось у X»
- **Если у клиента нет конкретных кейсов с цифрами** — НЕ выдумывать. Использовать только регалии (опыт лет, число клиентов, общие цифры завода/агентства). В HTML-цепочке добавить warning-карточку: «конкретных кейсов от клиента нет, обновим до v2 после получения». Запрашивать кейсы у клиента отдельно

## Не выдумывать конкретику продукта

Правило выше про кейсы распространяется на **любую конкретику продукта/услуги**, которой нет в брифе:
- **Цифры эффективности** (например «закрываем 80% вакансий», «срок подбора 14 дней», «удержание 90% после испытательного») — если клиент не дал, НЕ писать.
- **Модус работы услуги** (как именно агентство ищет, как сравнивает, как ведёт замену) — НЕ додумывать «риторически логичные» механики. Их даёт только клиент.
- **Критерии отбора, методология, фишки процесса** — то же самое.
- **Расширение списка ролей/услуг** за пределы того, что в брифе. Если в брифе «сборщик, контролер ОТК, инженер» — не дописывать «бригадиры, технологи, мастера», даже если кажется логичным.
- **Причины оттока клиентов / частотность их проблем.** «Самое частое, с чем у нас раньше клиенты спотыкались — X» — НЕ писать, если у клиента нет статистики оттока. Получатель прочитает это как факт о компании, а не как риторику.
- **Варианты ответов получателя в его уста.** «База не подошла по качеству, не хватило фильтров, цена смутила?» — этот перечень превращается в фактическое утверждение «вот по этим причинам обычно уходят». Если реальных причин в брифе нет — задавать вопрос без перечня.
- **Способы работы / процессы получателя.** «Через сайты, выписки, парсинг, готовые подписки» — мой перечень того, как ЦА может делать работу. Если в брифе/созвоне нет данных о реальных способах — задавать вопрос без перечня.
- **Этапы внутренних процессов получателя.** «В ручном сборе много времени уходит на проверку email и поиск ЛПР» — гипотезы про этапы работы получателя. Не писать, если нет источника.
- **Длительность встреч / звонков** («15 минут на разбор», «полчаса на демо») — если клиент не назвал стандартную длительность, не зашивать конкретное число. «Короткий звонок» — норм.
- **UX/функционал кабинета или сайта продукта** («там можно собрать выборку самостоятельно», «в кабинете есть фильтр X») — НЕ писать, если этот UX не подтверждён клиентом или скриншотом. Получатель проверит и не найдёт — доверие сгорит.
- **Допущения про админку клиента** («Вижу по кабинету, что вы пробовали», «По нашим логам вы заходили N раз»). Если клиент не сказал, что у него есть такая видимость по сегменту, не приписывать ему эту способность.

Допустимо: общие утверждения из брифа («подбираем под конкретную роль», «работаем с производствами 300-600 млн по РФ»), регалии головного бренда (Лига, агентство), плейсхолдеры подниш `[вариант 1 / вариант 2]`.

Если без конкретики письмо проседает — не закрывать пробел фантазией. Поставить открытое поле в warning-карточку и запросить у клиента отдельно.

## Pre-flight по содержанию (до Write, не после)

Grep-чеклист ниже ловит **форму** (ё, тире, антитезы, темы). Он НЕ ловит выдуманные смысловые утверждения. До записи цепочки в файл — обязательная процедура аудита содержания:

1. **Выписать списком все конкретные утверждения каждого письма.** Каждое предложение, где есть цифра, факт, причина, перечисление, длительность, описание процесса или UX — отдельной строкой.
2. **Для каждого утверждения пометить источник:**
   - ✓ **Бриф клиента** (с указанием поля)
   - ✓ **Созвон/исходники** (с цитатой)
   - ✓ **Кейс/PDF** (с указанием раздела)
   - ✓ **Лендинг/пресс-релиз** (с URL)
   - ✗ **Мой домысел / здравый смысл** — без источника
3. **Все ✗ — обработать перед Write:**
   - либо вырезать;
   - либо переформулировать в общую фразу без конкретики;
   - либо вынести в warning-карточку и запросить у клиента;
   - НЕЛЬЗЯ оставлять как есть «потому что логично».
4. Только после этого — Write.

Это правило важнее grep-чеклиста по форме: грамматическая ошибка ловится клиентом за секунду и правится, выдуманный факт превращается в подрыв доверия и не виден на проверке без специального аудита.

## Дедлайн как факт регламента ≠ тайм-оффер

Если на лендинге есть **реальный регламентный дедлайн** (анкета Службы Безопасности к дате X, лимит мест по регламенту предприятия, дата закрытия регистрации) — его можно использовать в дожимном письме как **естественный крючок срочности**:
- «Регистрация закрывается 4 июня — после этой даты допуск на производство по регламенту не оформят».
- Подаётся как факт регламента, а не как «успейте до...».
- Это не оферт, не скидка, не тайм-оффер. Юридика чистая (не нарушает запрет на тайм-офферы из юридики проекта).

Запрещено маскировать под регламент придуманную срочность («осталось 3 места», «цена вырастет после X») — если этого нет в лендинге, не писать.

## Кросс-сейл смежных событий в дожиме — только при совпадении ЦА

В дожимном письме (№2) может быть полезен кросс-сейл — упоминание других ближайших мероприятий клиента через двоеточие («останется контакт: 3 июня — Сима-ленд, 11-13 июня — Аскона, 18-19 июня — Ракета»). Но:
- **Совпадение ЦА обязательно.** Кросс-сейл уместен между событиями для одной аудитории (например, между миссиями для собственников 500+ млн). Между разными ЦА (например, миссии 500+ млн и HR-продукт для 300-600 млн) — НЕ ставить: получатель из мелкой ЦА не потянет более крупное событие, и кросс-сейл только размывает фокус основного CTA.
- **Один CTA на письмо.** Кросс-сейл = опциональный «остаётся контакт», не альтернативный CTA. Основной шаг должен оставаться один.
- **Только реальные ближайшие события** из календаря клиента, без выдумывания.

## Услуговые/постоянные продукты ≠ мероприятия

Если у клиента продукт-услуга с постоянным потоком заявок (HR-агентство, аутсорс, абонентка) вместо разового мероприятия:
- Структура «Приглашаю → Дата → Место → Программа → CTA» **не работает** — нет даты и места.
- Использовать **квалифицирующий вопрос** (правило №5) как основной заход письма №1: «У вас сейчас открыта вакансия [роль]?» / «Есть ли задача по X?».
- Тон **информационный** (правило №6): «пишу от агентства / запустили направление», без «я знаю вашу проблему».
- Регалии — головного бренда, а не нового продукта (если у продукта нет своего трекрекорда).
- Письмо №2 — дожим через вторую боль (если есть в брифе) + повтор CTA. Без кросс-сейла на смежные мероприятия, если ЦА не совпадает.

## Правила распределения болей по письмам

- **В каждом письме цепочки — своя ключевая боль.** Не повторять одну и ту же боль или близкие формулировки в разных письмах. Распределить разные боли из брифа по письмам №1, №2, №3
- Боли в №1, №2, №3 должны быть разными по сути (не «не хватает скорости» в №1 и «перенастройка съедает время» в №2 — это про одно и то же), а не просто переформулированными

## Плейсхолдеры подниш в письме №1

- Если сегмент покрывает несколько подниш и у каждого получателя своя — использовать формат `[вариант 1 / вариант 2 / вариант 3]` в квадратных скобках через слеш
- Оператор перед отправкой выбирает один вариант и оставляет в письме
- Все варианты — реальные подниши из брифа сегмента (не выдуманные)

## Таблица распределения в HTML-цепочке

- Перед блоком «Источники» в HTML-цепочке размещать таблицу-сетку
- Строки — письма (№1, №2, №3); колонки — **Боль / Регалии / УТП продукта**
- Это визуальный контроль, чтобы ничего не дублировалось между письмами одной цепочки и между цепочками одного клиента
- Таблица помогает на этапе проверки сразу увидеть, не повторяются ли формулировки

## Правила представления агентства

- **Крупные клиенты** — при первом знакомстве упоминать узнаваемые бренды из портфолио (Газпром, Кофемания, Мать и Дитя и т.д.) — дает вес
- **Регалии и преимущества — не повторять одни и те же** в разных письмах одной цепочки. Варианты: опыт (лет), конкретные бренды-клиенты, экспертиза в нише, работа по KPI, 80% клиентов работают 1+ год, не ведем конкурентов в одном гео
- **Избегать размытых формулировок** — «похожая ситуация» (с чем похожая?), «насколько это вам подходит» (что подходит?) — заменять на конкретику

## Правила работы между сегментами одного клиента

- **Уникальные сегменты — пишутся максимально по-новому.** Если сегмент имеет свои уникальные боли, угол захода, продукт — НЕ копировать формулировки из других цепочек этого же клиента. Менять старт письма (айсбрекер), заход к боли, главный УТП-аргумент, формулировки для перехода между блоками. Анализировать тщательно, не лениться.
- **Что должно отличаться между уникальными сегментами:** первая фраза письма, формулировка перехода к боли («самое частое, с чем приходят» / «частая жалоба» / «частая история» — варьировать), сами боли (брать разные из брифа), главные УТП-аргументы под продукт, цифры и кейсы.
- **Что можно оставлять неуникальным** (служебные шаблоны цепочек): «Возвращаюсь к прошлому письму» во 2-м письме, «Напомню о себе коротко – и больше не буду» в 3-м, базовая структура айсбрекер→боль→решение→CTA, плейсхолдеры `[приветствие]` / `[подпись]`, технические факты и регалии завода (но обвязка вокруг них должна быть разная).
- **Правило применимо и к разным клиентам в одной нише** — если делали цепочку для агентства А в нише Х, для агентства Б в той же нише Х фразы и заходы должны отличаться.
