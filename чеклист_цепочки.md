# Чек-лист идеальной цепочки

**Единый функциональный гейт качества.** Этот файл — не пересказ правил, а операционный список проверок, выведенный из них. Его проходят ДВА агента независимо (правило 4):

- **Копирайтер** — вслух в чате перед каждой записью в `цепочки/*.html` (self-check). Не «прочитать молча», а явно перечислить пункты с ✓/✗.
- **Ревьюер** — свежим взглядом после Write, **по осям A-E раздельно** (дробление, правило 1): каждый проход берет одну ось, чистый контекст, по каждому пункту вердикт ✓/✗ с местом.

Чеклист держать в синхроне с правилами: меняется skill/CLAUDE.md → правится этот файл. Рассинхрон = инструмент проверки врет.

> Источники правил по каждой оси — в skill `аутрич-стратегия`, skill `аутрич-цепочка`, `справочники/формат-цепочек.md`, `CLAUDE.md`. Чеклист только перечисляет, что проверить, без дублирования логики (правило 2).

## Ось A — Факты и источники

Красные линии 1 и 2. Самый тяжелый по последствиям блок — выдуманный факт подрывает доверие.

- **Source-check листом:** все конкретные утверждения каждого письма выписаны построчно
- Каждое утверждение: ✓ источник (с указанием поля/цитаты/раздела) или ✗ домысел
- Все ✗ обработаны ДО Write: вырезаны / в общую фразу / в warning. Ни одного «оставил, потому что логично»
- Каждая цифра привязана к источнику в блоке «Источники» под цепочкой
- Цифра из кейса → атрибутирована компании в тексте («Билайн +24%»)
- Цифра из общего брифа → «в среднем по проектам», без конкретной компании
- Цифра с лендинга → как речь спикера/тезис, не как наш факт
- **Особо рисковые категории домысла проверены отдельно:**
  - Цифры эффективности (% / срок / кратность / удержание)
  - Модус работы услуги, критерии отбора, методология, фишки процесса
  - Расширение списка ролей/услуг за пределы брифа
  - Причины оттока клиентов / частотность их проблем
  - Варианты ответов получателя, вложенные в его уста
  - Способы работы получателя; этапы его внутренних процессов
  - Длительность встреч/звонков (число)
  - UX/функционал кабинета или сайта; допущения про админку
- Кейсы: авторство ясно («мы сделали для X», не «получилось у X»)
- Нет кейсов с цифрами → не выдумано, регалии + warning «обновим до vN»
- Соцпруф клиентами — именами, без обобщений («крупные компании», «нам доверяют бренды» — запрещено)

## Ось B — Структура и распределение

skill `аутрич-цепочка`: роли писем, CTA-лестница, распределение, референс.

- **№1:** триггер/айсбрекер + знакомство + легкий вопрос. Без созвонов
- **№2:** разбор 2-3 нюансов сегмента + персонализированный артефакт (КП/расчет/визуализация под получателя)
- **№3:** обнуляющий follow-up или квалифицирующий вопрос на горизонт. Без повтора оффера №2, без давления
- **CTA-лестница растет:** №1 легкий → №2 средний → №3 тяжелый или обнуляющий
- В каждом письме — своя ключевая боль (разные по сути, не переформулированные)
- Регалии не повторяются между письмами одной цепочки
- В разных письмах — разные кейсы
- **Таблица «Боль / Регалии / УТП»** (строки №1/№2/№3) в HTML перед «Источниками» — визуальный контроль на дубли
- **Референс-кейс (если есть):** структура взята блок-в-блок (слой 2), не «по выводам», не пересказ. Порядок блоков, длина писем, тип захода, CTA — от референса
- Факты из самого референса НЕ переехали как наши без источника под нашего клиента (слой 3)

## Ось C — Стратегия и тон

skill `аутрич-стратегия` (включая новые: Хант, дуга боли) + skill `аутрич-цепочка` (модус, канцелярит).

- **Уровень Ханта определен ДО письма:** на каком уровне ЦА (1-2 Problem/Unaware vs 3-4 Solution/Product Aware)
- ЦА на 1-2 → заход с симптома, не с продукта; боль широкая, без конкретных сцен
- ЦА на 3-4 → Solution Aware заход рабочий
- **Дуга боли дожата до бизнес-следствия адресата:** триггер → техпричина → следствие в деньгах/времени. Нет обрыва на инженерной причине
- Самопроверка по каждой боли: «и что это значит для адресата?» — дописано
- **Модус в №1 раскрыт**, не абстрактный ярлык: 1-2 фразы «как работаем по шагам» из источника (если ниша требует)
- Триггер айсбрекера реальный (закон/кейс/сайт/гео/МП/вакансия/событие/стек)
- Регуляторный триггер — только при зафиксированном в брифе законе
- Без триггера — квалифицирующий вопрос + «почему спрашиваю»
- Сегмент узкий; для N сегментов — своя строка «почему именно вам» (бизнес-логика)
- Тон мимикрирует под живого менеджера; новые продукты — информационный тон
- Язык клиента (формулировки из звонков/брифов); без канцелярита («подсказать по выгоде» и т.п.)
- Скидка — «дарим», не «даем»

## Ось D — Форма и типографика

Красные линии 3, 4, 8 + `справочники/формат-цепочек.md`.

- **Тема ЕСТЬ в каждом из 3 писем** (строкой `Тема: ...` перед `[приветствие]`)
- Тема ОДИНАКОВАЯ во всех 3 письмах, с переменной `{Компания}`, общая по сегменту
- Приветствия/подписи — только `[приветствие]` / `[подпись]`, не текстом
- Без антитез «не X, а Y» / «вместо X — Y»
- «е» вместо «ё» везде
- Тире `–` (короткое) для текста; длинное `—` отсутствует
- Числовые диапазоны через `-` без пробелов («2-5 минут»)
- Ровно 3 письма (или N×3 в A/B); нумерация «Письмо №1/№2/№3»
- Согласование рода глаголов с подписью (женская → «увидела/нашла/благодарна»)
- HTML: line-height 1.15; Arial 11pt body + h2 18pt bold; inline-style на каждом блоке
- Кнопка copy на каждое письмо + «Копировать все»; маркер `<!-- PREFLIGHT-OK -->` на месте
- **5 grep'ов после Write — пусто:**
  - `grep -nE "Здравствуйте|Добрый день|доброгоВремениСуток|С уважением|^Подпись" f.html`
  - `grep -nE "не [А-Яа-я]+.*, а |, а не |вместо .* —" f.html`
  - `grep -nE "ё|Ё" f.html`
  - `grep -nE "[0-9]–[0-9]" f.html`
  - `grep -nE "—" f.html`
- 6-й grep (размытый соцпруф) — пусто И проверено глазами каждое предложение про клиентов

## Ось E — Спецблоки и полнота

PRE-FLIGHT, warning, A/B, кросс-сейл, дедлайн, плейсхолдеры.

- **PRE-FLIGHT блок был выведен в чат** ДО Write: source-check лист + таблица распределения + CTA-лестница + разводка сегментов
- **A/B-дефолт:** два неисключающих захода собраны как «Вариант A / Б», а не вынесены в вопрос «какой выбрать»
- `AskUserQuestion` по вариантам письма не использован (вопрос — только операционная развилка, которую знает один пользователь)
- Без сегментов / без отраслевой привязки — A/B из N цепочек, боли и заходы между цепочками не пересекаются
- Кросс-сейл смежных событий — только при совпадении ЦА; один CTA на письмо
- Дедлайн как регламент — только если есть в лендинге клиента (не выдуманная срочность)
- Услуговые/постоянные продукты — без «Приглашаю→Дата→Место», через квалифицирующий вопрос
- Плейсхолдеры подниш `[вариант 1 / вариант 2]` — все варианты реальные из брифа
- PS «это не AI» — только в IT/digital; PS «пишу на несколько почт» — только в №2/№3
- **Warning-карточка перечисляет ВСЕ открытые поля для клиента**; все ✗ из source-check ушли в warning, не в письма
- **Кросс-сегментная гигиена:** между сегментами одного клиента первая фраза, переход к боли, главный УТП — разные. Служебные шаблоны можно повторять
- Файл открыт в Chrome; статус и путь в MEMORY.md

## Как дробится ревью (правило 1)

Ревьюер не проходит весь чеклист одним проходом «по памяти» — он берет **оси A-E как отдельные узкие задачи**. Оркестратор может гонять их параллельно (каждая ось — чистый контекст, фокус только на своих пунктах, минимум пропусков), затем свести вердикты:

- **Проход A** — факты/источники: сверить каждую цифру и каждое конкретное утверждение с карточкой проекта / кейсами / лендингом.
- **Проход B** — структура: роли, лестница, распределение без повторов, наложение референса.
- **Проход C** — стратегия/тон: уровень Ханта, дуга боли, модус, мимикрия.
- **Проход D** — форма: 5 grep'ов + 6-й + типографика + тема в каждом письме.
- **Проход E** — полнота: PRE-FLIGHT, warning, A/B, спецблоки.

Каждый проход возвращает: **блокеры** (с местом и правкой) + **замечания** + «что проверил и чисто». Сводный вердикт — годна / на доработку — выносится после всех осей.
