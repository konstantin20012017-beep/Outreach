# Чек-лист идеальной цепочки

**Обязательно проходить вслух в чате перед каждой записью в `цепочки/*.html`.** Не «прочитать молча», а явно перечислить пункты с отметкой — иначе процедура не сработает. См. CLAUDE.md раздел «Pre-flight по содержанию».

## 0. Подготовка

- Все материалы (чат, файлы, ссылки) перечислены и обработаны до начала (Правило №0.5)
- PDF/DOCX/XLSX открыты через pdftotext/pandoc/openpyxl; если не выходит — явно сказано клиенту
- Бриф `брифы/<проект>_бриф.md` + `.html` создан/обновлён (Правило №0)
- Сырьё в `брифы/<проект>_исходники.md`
- Бриф `.html` открыт в Chrome
- MEMORY.md обновлён

## 1. Стратегия

- Сегмент узкий
- Для N сегментов — своя строка «почему именно вам» (бизнес-логика)
- Без сегментов — A/B-тест из N цепочек, боли не пересекаются между цепочками
- Референс клиента — копируется блок-в-блок, не «по выводам»

## 2. Структура писем

- №1: айсбрекер + знакомство + лёгкий вопрос. Без созвонов
- №2: разбор нюансов + персонализированный артефакт
- №3: обнуляющий follow-up или квалифицирующий вопрос. Без повтора оффера №2
- CTA-лестница растёт: №1 лёгкий → №2 средний → №3 либо тяжёлый, либо обнуляющий

## 3. Айсбрекер №1

- Триггер реальный (закон/кейс/сайт/гео/МП/вакансия/событие/стек)
- Регуляторный триггер — только при зафиксированном в брифе законе
- Без триггера — квалифицирующий вопрос + «почему спрашиваю»

## 4. Распределение по письмам

- В каждом письме своя ключевая боль
- Регалии не повторяются между письмами одной цепочки
- В разных письмах разные кейсы
- Таблица **Боль / Регалии / УТП** в HTML перед «Источниками»

## 5. Цифры и факты (Правило №0.6)

- Каждая цифра привязана к источнику в блоке «Источники»
- Цифра из кейса → к компании в тексте
- Цифра из общего брифа → «в среднем по проектам», без конкретной компании
- Цифра с лендинга → как речь спикера, не как наш факт

## 6. Source-check содержания (до Write)

- Все конкретные утверждения выписаны построчно
- Каждое: ✓ источник или ✗ домысел
- Все ✗ обработаны: вырезаны / в общую фразу / в warning
- Особо рисковые категории проверены:
  - Причины оттока клиентов / частотность их проблем
  - Варианты ответов получателя в его уста
  - Способы работы получателя
  - Этапы внутренних процессов получателя
  - Длительность встреч/звонков
  - UX/функционал кабинета продукта
  - Допущения про админку клиента
- Цифры эффективности, модус услуги, критерии отбора, расширенные списки ролей — только из брифа

## 7. Тон

- Мимикрирует под живого менеджера
- Новые продукты — информационный тон, не «я знаю вашу проблему»
- Язык клиента (формулировки из звонков/брифов)
- Без канцеляризмов («подсказать по выгоде» и т.п.)
- Без антитез (`не X, а Y` / `вместо X — Y`)
- Скидка — «дарим», не «даём»

## 8. CTA

- Соответствует длине цикла сделки
- Скидка/КП не закрываются сразу в №1
- Цель — звонок/мессенджер
- «Примеры работ» в визуальных нишах — наравне со скидкой или впереди
- PS «это не AI» — только в IT/digital-нишах
- PS «пишу на несколько почт» — только в №2 или №3

## 9. Спецблоки

- Кросс-сейл смежных событий — только при совпадении ЦА
- Дедлайн как регламент — только если есть в лендинге клиента
- Услуговые/постоянные продукты — без «Приглашаю→Дата→Место», через квалифицирующий вопрос
- Плейсхолдеры подниш `[вариант 1 / вариант 2]` — все варианты реальные из брифа

## 10. Кейсы

- Авторство ясно: «мы сделали для X», не «получилось у X»
- Нет конкретных кейсов с цифрами → не выдумывать, регалии + warning

## 11. Формат HTML

- Темы письма нет
- Только `[приветствие]` / `[подпись]`
- «ё» → «е»
- Тире `–`, не `—`
- Числовые диапазоны через `-` без пробелов
- Ровно 3 письма (или N×3 в A/B-тесте)
- Нумерация «Письмо №1/№2/№3»
- line-height: 1.15
- Arial 11pt body + h2 18pt bold
- Inline-style на каждом блоке (под Google Docs paste)
- Кнопка copy на каждое письмо + «Копировать все»

## 12. Pre-flight grep (после Write)

- `grep -nE "Тема:|Здравствуйте|Добрый день|С уважением|^Подпись" file.html` — пусто
- `grep -nE "не [А-Яа-я]+.*, а |, а не |вместо .* —" file.html` — пусто
- `grep -nE "ё|Ё" file.html` — пусто
- `grep -nE "[0-9]–[0-9]" file.html` — пусто
- `grep -nE "—" file.html` — пусто
- Если подпись женская — grep по «увидел/нашел/благодарен/готов/рад»

## 13. Кросс-сегментная гигиена

- Между уникальными сегментами одного клиента: первая фраза, переход к боли, главный УТП-аргумент — разные
- Служебные шаблоны («Возвращаюсь к прошлому письму», «Напомню коротко – и больше не буду») — можно повторять
- Между клиентами в одной нише — заходы разные

## 14. Финал

- Файл открыт в Chrome
- Warning-карточка перечисляет все открытые поля для клиента
- Все ✗ из source-check — в warning, не в письмах
- Статус и путь к файлу в MEMORY.md
